# OrbitFabric Studio v0.3.0 Release Notes

Release: `v0.3.0 - Contract Navigation Surface`

Status: release candidate

---

## Summary

OrbitFabric Studio v0.3.0 extends the implemented product loop from:

```text
Open -> Inspect -> Validate -> Understand
```

to:

```text
Open -> Inspect -> Validate -> Navigate
```

This release adds read-only Contract Navigation based on OrbitFabric Core-owned derived reports.

Studio consumes Core `model_summary.json` and `entity_index.json` reports.

Studio does not parse Mission Model YAML semantically.

Studio does not infer entities, relationships, graph edges, source locations or YAML AST nodes privately.

OrbitFabric Core remains authoritative for mission semantics.

---

## Added

- Fixed backend command for Core `export model-summary`.
- Fixed backend command for Core `export entity-index`.
- Studio-controlled report paths for Core model summary and entity index exports.
- TypeScript models for Core `model_summary.json`.
- TypeScript models for Core `entity_index.json`.
- Minimal read-only report parsers for Core lint, model summary and entity index reports.
- Frontend action for fixed Core `export model-summary`.
- Frontend action for fixed Core `export entity-index`.
- Core-derived Contract Domains panel from `model_summary.json`.
- Core-derived Contract Entities panel from `entity_index.json`.
- Core-derived domain index summary with indexed and not-indexed states.
- Entity records grouped by Core-reported domain.
- Safe read-only source-file links from Core model summary domain records.
- Safe read-only source-file links from Core entity index domain and entity records.
- Missing or unsupported Core model summary fallback text.
- Missing or unsupported Core entity index fallback text.

---

## Core surfaces consumed

v0.3.0 consumes these Core-owned surfaces:

```text
orbitfabric export model-summary <mission_dir> --json <studio_report_path>
orbitfabric export entity-index <mission_dir> --json <studio_report_path>
```

Structured reports:

```text
kind: orbitfabric.model_summary
summary_version: 0.1

kind: orbitfabric.entity_index
index_version: 0.1
```

Tested with OrbitFabric Core `0.8.2`.

```text
examples/demo-3u
```

Observed model summary result:

```text

Result: PASSED
```

Observed entity index result:

```text
OrbitFabric Entity Index Export 0.8.2
Mission: demo-3u
Model version: 0.1.0
Total entities: 46
Result: PASSED
```

Validated UI behavior:

```text
Run --version
Run inspect mission
Run lint mission
Run export model-summary
Run export entity-index
Display Contract domains
Display Contract entities
Display indexed and not-indexed domain states
Display mode_transitions and policies without synthetic entity records
Open Core-indicated source files read-only
Preserve stdout, stderr and exit code
```

---

## Not included

v0.3.0 intentionally does not include:

- editing;
- visual model editing;
- semantic YAML parsing;
- private entity extraction;
- private domain registry when Core `model_summary.json` is available;
- private relationship resolution;
- graph rendering;
- relationship navigation;
- dependency graph;
- source line or column navigation;
- fake source spans;
- quick fixes;
- suppressions;
- scenario runner;
- generator workflow beyond fixed Core-owned report exports;
- ground artifact explorer;
- runtime artifact explorer;
- plugin UI;
- arbitrary command execution;
- arbitrary OrbitFabric CLI argument entry;
- mission-control UI;
- live telemetry;
- command uplink.

---

## Known limitations

- Studio displays source-file links only when the Core-provided file reference matches a known source model file in the workspace.
- Studio does not provide line or column jumps because Core does not provide line or column metadata in these reports.
- Studio does not expose relationships because Core v0.8.2 does not expose a relationship manifest, relationship graph or dependency graph.
- Older Core compatibility is based on fixed command behavior and report availability, not only version-string parsing.

## Release gate

Before tagging `v0.3.0`, the following local checks should be completed:

```bash
npm run build
source "$HOME/.cargo/env"
npm run tauri:dev
```

The release metadata files must be updated consistently before tagging:

```text
package.json
package-lock.json
src-tauri/Cargo.toml
src-tauri/Cargo.lock
src-tauri/tauri.conf.json
```

Lockfiles should be refreshed by the local package managers rather than edited by hand.
