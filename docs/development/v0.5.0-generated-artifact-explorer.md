# OrbitFabric Studio v0.5.0 Generated Artifact Explorer

Status: planning baseline  
Milestone: `v0.5.0 - Generated Artifact Explorer`  
Nature: read-only generated artifact inspection slice

---

## Purpose

OrbitFabric Studio v0.5.0 introduces a read-only Generated Artifact Explorer.

The goal is to let a user open an OrbitFabric workspace and inspect generated outputs already present in the workspace without changing files, generating new outputs or reconstructing Mission Data Contract semantics inside Studio.

Studio remains downstream from OrbitFabric Core.

OrbitFabric Core remains authoritative for Mission Model loading, validation, scenario evidence, generated artifacts, Core-owned structured surfaces and engineering meaning.

---

## Product Loop

The v0.5.0 planning loop is:

```text
Open -> Inspect -> Validate -> Navigate -> Explain Relationships -> Inspect Generated Artifacts
```

The Generated Artifact Explorer answers:

```text
Which generated artifacts are present?
Where are they located?
What broad artifact class do they belong to?
Are they known Core-documented outputs or unknown generated files?
Can I inspect their raw content read-only?
```

It does not answer:

```text
What mission semantics should be inferred from an artifact?
What operational action should be taken?
What runtime behavior follows?
What ground behavior follows?
How should the Mission Model be changed?
How should generated code be merged with user code?
```

---

## Baseline Inputs

v0.5.0 starts from Studio v0.4.0.

The verified v0.4.0 baseline already provides:

- local workspace opening;
- structural Mission Model file detection;
- scenario source detection;
- generated location detection;
- read-only text viewer;
- fixed Core command execution for known commands;
- Core lint report display;
- Core model summary display;
- Core entity index display;
- Core relationship manifest display.

v0.5.0 must not weaken any v0.4.0 boundary.

## Artifact Classes

The initial v0.5.0 artifact classes are conservative.

```text
reports
  Core-owned structured reports
  validation reports
  scenario JSON evidence reports

logs
  scenario timeline logs

docs
  generated Markdown mission documentation
  generated data-flow documentation

runtime
  runtime-facing contract manifests
  generated C++17 contract bindings
  generated C++17 host-build smoke files

ground
  ground-facing contract manifests
  JSON ground dictionaries
  CSV ground dictionaries
  human-reviewable ground Markdown artifacts

unknown
  any generated file under generated/ that is not recognized by conservative rules
```

Unknown does not mean invalid.

Unknown only means Studio does not have a documented classifier for that artifact in this slice.

## Planned Capabilities

v0.5.0 should add the following capabilities in small PRs:

- define a generated artifact inventory model;
- inspect `generated/` recursively through a controlled backend command;
- classify files by conservative path, extension and known Core-documented names;
- group artifacts by broad class;
- show path, file name, extension, size and detected class;
- distinguish known artifacts from unknown generated files;
- open supported text artifacts in the existing read-only viewer;
- show a read-only detail panel for the selected artifact;
- display provenance only when it is directly derivable from documented Core path conventions or manifest fields already present in the artifact;
- preserve raw file preview without semantic interpretation.

---

## Non-goals

v0.5.0 must not add:

```text
editing
generated file modification
Mission Model modification
arbitrary shell command execution
arbitrary OrbitFabric CLI argument entry
new generator workflow
scenario runner
runtime execution
build-system integration
flight software claim
ground runtime claim
mission-control UI
live telemetry
command uplink
private YAML semantic parsing
private Mission Model validation
private artifact semantics
private dependency graph
private relationship graph
plugin execution
plugin discovery
```

---

## Implementation Notes

The first implementation PR after this planning baseline should add only the TypeScript model for generated artifact inventory.

Backend recursive inspection should be a later PR.

The backend should read only inside the selected workspace and should stay bounded.

The read-only viewer limit and supported text file logic should remain conservative.

Binary files should be listed but not previewed unless a future explicit design adds safe support.

---

## Exit Criteria

v0.5.0 is complete only when Studio can:

1. inspect generated artifacts already present in the selected workspace;
2. group artifacts into conservative classes;
3. distinguish known Core-documented artifacts from unknown generated files;
4. show metadata without modifying files;
5. open supported text artifacts read-only;
6. preserve source, derived and generated boundaries;
7. avoid generating artifacts, running arbitrary commands or inferring Mission Model semantics.
