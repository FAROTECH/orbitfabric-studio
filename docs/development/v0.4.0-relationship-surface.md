# OrbitFabric Studio v0.4.0 - Relationship Surface

This document records the planned implementation slices for `v0.4.0 - Relationship Surface`.

The purpose of v0.4.0 is to make Core-owned mission contract relationships inspectable in Studio by consuming OrbitFabric Core `relationship_manifest.json`.

Studio v0.4.0 is downstream from OrbitFabric Core.

Studio must not become a Mission Model parser, relationship resolver, dependency graph engine, runtime interpreter, ground behavior interpreter or second Core.

---

## Baseline

Studio v0.3.0 implements the Contract Navigation Surface.

It already consumes:

```text
Core model_summary.json
Core entity_index.json
```

It answers:

```text
What contract domains are present in this mission, and where can I inspect them?
What contract entities are defined in this mission, and where can I inspect their source file?
```

It does not yet answer:

```text
How are these entities related?
```

OrbitFabric Core v1.0.0 provides the required Core-owned relationship surface:

```text
relationship_manifest.json
kind: orbitfabric.relationship_manifest
manifest_version: 0.1-candidate
status: candidate
```

The fixed Core command is:

```text
orbitfabric export relationship-manifest <mission_dir> --json <report_path>
```

The default Core output path is:

```text
generated/reports/relationship_manifest.json
```

---

## Product Loop

v0.4.0 extends the implemented loop from:

```text
Open -> Inspect -> Validate -> Navigate
```

to:

```text
Open -> Inspect -> Validate -> Navigate -> Explain Relationships
```

The shorter form is also acceptable in UI copy:

```text
Open -> Inspect -> Validate -> Navigate -> Explain
```

---

## Question Answered

Studio v0.4.0 answers:

```text
How are indexed mission contract entities related, according to Core?
```

Studio v0.4.0 does not answer:

```text
How should the system execute those relationships?
What runtime behavior follows from them?
What ground behavior follows from them?
What is the dependency graph?
What is the live operational state?
What should be edited?
```

---

## Core Surface Consumed

Studio v0.4.0 consumes only the Core-owned Relationship Manifest Surface.

Expected report identity:

```text
kind: orbitfabric.relationship_manifest
manifest_version: 0.1-candidate
status: candidate
```

Expected top-level report areas:

```text
manifest_version
kind
orbitfabric_version
status
mission
source
boundaries
counts
relationship_types
relationships
derivation_policy
```

Studio must not invent fields that Core does not expose.

Studio must not add line, column, source span, YAML AST, dependency graph, graph layout, plugin or runtime fields to the report model.

---

## Boundary Labels

The UI must make the following boundaries visible:

```text
Core relationship manifest
not relationship graph
not dependency graph
no source locations
no runtime behavior
no ground behavior
```

These labels are not cosmetic.

They protect the Core and Studio separation.

---

## Relationship Families

The initial v0.4.0 implementation may display relationship families admitted by Core v1.0.0.

Studio must treat the relationship type list as report data.

Studio must not create a private registry of relationship semantics.

Studio may display:

```text
relationship_type
display_name
from_domain
to_domain
derived_from.model_field
relationship_count
```

Studio must not add relationship types that are missing from the Core report.

---

## Relationship Records

Studio may display Core relationship records with:

```text
relationship_id
relationship_type
from.domain
from.id
to.domain
to.id
derived_from.model_field
```

Each visible relationship record must come directly from Core `relationship_manifest.relationships`.

Studio must not infer additional records from naming convention, string similarity, ID prefixes, source file names, YAML ordering, YAML formatting, generated Markdown, generated runtime files, generated ground files, stdout, stderr or UI state.

---

## Entity Linking

Studio may link relationship endpoints to already indexed entities when `entity_index.json` is available and compatible.

Allowed rule:

```text
relationship endpoint -> matching entity_index entity by domain and id
```

If the entity index is not available, Studio must still show the relationship record but without entity detail links.

Studio must not create synthetic nodes.

Studio must not create synthetic edges.

Studio must not infer missing entities.

---

## Explanation Panel

The relationship explanation panel must be read-only.

For a selected relationship record, it should explain:

```text
this relationship comes from Core relationship_manifest.json
it is derived from the explicit Mission Model field shown in derived_from.model_field
it is not inferred by Studio
it does not represent runtime behavior
it does not represent ground behavior
it does not represent a dependency graph
```

A read-only source file link may be shown only when it is safely resolvable from `entity_index.json` or from an already known Core-provided source file.

No line or column jump is allowed in v0.4.0.

## Deferred Slice - Optional Record Visualization

A visual graph-like rendering is deferred unless it can be kept strictly controlled.

If introduced later:

```text
each visible edge must correspond one-to-one to a Core relationship record
each visible node must correspond to an entity_index entity
layout must have no semantic meaning
Studio must not calculate extra edges
Studio must not call it a dependency graph
Studio must not call it a relationship graph engine
```

This slice may be moved to v0.5.0 or later if it risks scope creep.

## Version Compatibility Target

Studio v0.4.0 targets Core v1.0.0 behavior for the Relationship Manifest Surface.

Compatibility must be behavior-based:

```text
command succeeds
report exists
report kind is orbitfabric.relationship_manifest
manifest_version is recognized
required top-level fields are present
```

Version strings may be displayed, but they must not be the only compatibility signal.

```text
run the fixed Core relationship-manifest export command
```
