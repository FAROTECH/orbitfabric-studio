# OrbitFabric Studio v0.1.0 — Development Slices

This document records the implementation slices for `v0.1.0 — Read-only Mission Project Viewer`.

The purpose of v0.1.0 is to establish a runnable desktop application that can open and inspect an OrbitFabric workspace without editing it and without becoming a second model engine.

---

## Slice 1 — Minimal application scaffold

Implemented:

```text
Tauri 2 desktop shell
React frontend
TypeScript configuration
Vite build setup
Minimal static Studio shell screen
Minimal Tauri backend entrypoint
Minimal default Tauri capability
```

The scaffold established the application baseline before real workspace inspection began.

---

## Slice 2 — Workspace opening and structural inspection

Implemented:

```text
local directory selection
structural workspace inspection command
mission directory detection
expected Mission Model file detection
scenario YAML file detection
generated location detection
source / scenario / derived / generated labels
read-only inspection UI
```

This slice remains structural only.

It does not perform OrbitFabric Core validation.

It does not parse Mission Model semantics.

It does not infer entity relationships.

It does not generate artifacts.

It does not modify workspace files.

---

## Slice 3 — Read-only file viewer

Implemented:

```text
bounded read-only text file loading
Monaco Editor read-only display
YAML, JSON, Markdown, text/log and C/C++ text artifact support
workspace containment check
maximum text file size guardrail
UTF-8 text-only loading
```

The file viewer exists to inspect already-detected source files and text artifacts.

It does not edit files.

It does not validate file contents.

It does not infer model semantics from YAML.

It does not create a private Studio model.

---

## Slice 4 — Controlled OrbitFabric Core command status

Implemented:

```text
fixed OrbitFabric --version command
fixed OrbitFabric inspect mission command
raw stdout / stderr / exit-code display
user-configurable executable path
no arbitrary argument entry
no shell execution through frontend permissions
```

This slice introduces controlled Core process invocation only as a status and observability bridge.

It does not run lint.

It does not generate artifacts.

It does not execute scenarios.

It does not interpret command output as validation diagnostics.

It does not create a Studio-owned semantic model.

---

## Permission boundary

The current Tauri capability grants:

```text
core:default
dialog:allow-open
```

The dialog permission is used only to allow the user to select a local directory.

No broad frontend filesystem permission is granted.

Workspace traversal is performed by the dedicated Rust command:

```text
inspect_workspace
```

Read-only text loading is performed by the dedicated Rust command:

```text
read_text_file
```

`read_text_file` requires the requested file to remain inside the selected workspace.

Controlled Core command execution is performed by dedicated Rust commands:

```text
run_core_version
run_core_inspect_mission
```

These commands expose only fixed argument sets.

Shell execution permissions are not granted.

---

## Architecture boundary

The workspace inspection command may classify files and directories structurally.

It may report:

```text
source model files
scenario source files
derived report locations
generated output locations
missing expected source filenames
warnings about structural detection
```

The read-only file viewer may display text file contents.

The Core command status panel may display raw process output.

It must not report:

```text
model validity
semantic correctness
entity relationships
lint status
scenario evidence status
artifact freshness
Core compatibility
```

Those remain OrbitFabric Core concerns.

OrbitFabric Core remains authoritative.

The Mission Model remains the source of truth.

Studio remains downstream.
