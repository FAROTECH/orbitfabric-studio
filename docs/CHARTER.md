# OrbitFabric Studio — Project Charter

## 1. Purpose

OrbitFabric Studio exists to make OrbitFabric Mission Data Contracts inspectable, navigable and understandable through a local-first visual engineering workbench.

Studio does not replace OrbitFabric Core.
Studio does not replace the Mission Model.
Studio does not create independent mission semantics.

The OrbitFabric Mission Model remains the source of truth.

## 2. Core Statement

OrbitFabric Studio is not where mission semantics are created.
It is where mission semantics become inspectable.

## 3. Problem Statement

OrbitFabric Core is evolving from a model-first validation and generation framework into a richer Mission Data Contract layer covering multiple engineering domains.

As the contract grows, the model spans relationships among:

- spacecraft
- subsystems
- modes
- telemetry
- commands
- events
- faults
- payloads
- data products
- storage intent
- downlink intent
- contact windows
- commandability rules
- autonomy rules
- scenario evidence
- runtime-facing bindings
- ground-facing artifacts
- future plugins and extensions

These relationships can be validated, reported and generated by OrbitFabric Core, but they are not always easy to inspect as separate YAML files, command-line outputs, JSON reports and generated documentation.

Studio exists because the Mission Data Contract is becoming an engineering graph, not just a file set.

The purpose of Studio is to expose that graph without changing its authority model.

Studio is not limited to post-processing completed contracts. It starts from inspection and validation because those are the safest foundations, but its long-term scope includes assisted, controlled and validation-gated contract authoring.

## 4. Product Thesis

OrbitFabric Studio turns the Mission Data Contract into an inspectable engineering surface.

It helps users answer questions such as:

- What is defined in this mission?
- Is the contract valid?
- Which entities are connected?
- Which assumptions are explicit?
- Which generated artifacts came from this model?
- Which scenario evidence supports a given data-flow assumption?
- Which runtime-facing or ground-facing outputs can be consumed by implementation teams?

Studio must make these answers easier to reach without hiding the underlying contract.

## 5. Authority Model

The Mission Model is authoritative.

OrbitFabric Core is authoritative for:

- model loading
- schema interpretation
- semantic validation
- lint rules
- scenario execution
- scenario evidence
- generated documentation
- generated reports
- runtime-facing contract bindings
- ground-facing contract exports
- future plugin semantics

Studio is authoritative only for:

- user interface state
- visual organization
- navigation flows
- presentation of diagnostics
- presentation of generated artifacts
- local workspace interaction
- future controlled editing workflows, when explicitly supported

Studio may display engineering meaning.
Studio must not invent engineering meaning.

## 6. Dependency Direction

The dependency direction is strict:

```text
OrbitFabric Core -> machine-readable outputs -> Studio Adapter Layer -> Studio UI
```

OrbitFabric Studio may depend on OrbitFabric Core.

OrbitFabric Core must not depend on OrbitFabric Studio.

Studio must not force OrbitFabric Core semantics to change for UI convenience.

When Studio requires additional structured information, the preferred solution is to improve OrbitFabric Core outputs rather than duplicating core logic inside Studio.

## 7. Scope

Studio may eventually support:

- opening an OrbitFabric mission workspace
- inspecting mission model files
- invoking OrbitFabric validation
- displaying validation summaries and diagnostics
- visualizing relationships among contract entities
- inspecting generated Markdown documentation
- inspecting generated JSON reports
- inspecting runtime-facing bindings
- inspecting ground-facing artifacts
- inspecting scenario evidence
- exporting visual inspection snapshots
- supporting controlled, validation-gated editing for selected domains

Studio must evolve through narrow, verifiable slices.

The initial product path is:

```text
inspect -> validate -> navigate -> visualize -> inspect generated artifacts -> inspect evidence -> controlled editing
```

Editing is intentionally delayed until the read/inspect/validate loop is stable.

Studio is not read-only by identity.

Studio is read-only by initial maturity strategy.

## 8. Explicit Non-authority

Studio must not become:

- a second OrbitFabric Core
- an alternative Mission Model format
- an independent validator
- an independent scenario engine
- a flight software framework
- an OBC runtime
- a ground segment
- a mission control system
- a live telemetry dashboard
- a command uplink console
- a spacecraft simulator
- a CCSDS/PUS/CFDP implementation
- a Yamcs/OpenC3 replacement
- a cloud collaboration platform
- a generic YAML IDE
- a generic diagramming tool

These boundaries are not temporary limitations. They are part of the project identity.

## 9. Data and Artifact Principles

Studio must always distinguish between:

```text
source model      = authoritative user-authored Mission Model files
derived report    = OrbitFabric Core output derived from the source model
generated output  = disposable artifact generated from the contract
UI state          = local representation used by Studio
```

Generated artifacts must be clearly marked as generated.

Studio must not encourage users to manually edit generated outputs.

## 10. Validation Principle

Validation decides.

Studio may present validation results, organize diagnostics and make invalid relationships easier to locate.

OrbitFabric Core decides validity.

Studio must not silently downgrade, reinterpret, mask or override validation failures.

## 11. Assisted Authoring and Controlled Editing Principle

Studio starts as read/inspect/validate first.

This does not mean that Studio is only a validator of already-completed contracts.

The long-term role of Studio includes helping users write, extend and refine Mission Data Contracts through explicit, reviewable and validation-gated changes.

When assisted authoring or editing is introduced, it must follow this pattern:

```text
user action -> explicit model patch -> visible diff -> write -> OrbitFabric validation -> accepted/rejected state
```

No hidden auto-fixes.
No silent rewrites.
No unsupported domain mutation.
No semantic invention.
No internal Studio-only contract representation.

## 12. Local-first Principle

Studio is designed first as a local engineering workbench.

The initial workflow assumes local OrbitFabric mission workspaces and local invocation of OrbitFabric Core.

Cloud workspaces, user accounts, remote collaboration and hosted storage are out of scope unless justified by a future architecture decision.

## 13. Release Philosophy

Studio releases must be narrow, inspectable and aligned with OrbitFabric Core maturity.

Each release must define:

- what it enables
- which OrbitFabric Core surfaces it consumes
- which capabilities are intentionally excluded
- which outputs are source, derived or generated
- which parts are experimental

Studio must not expose polished workflows for unstable core surfaces unless they are explicitly marked experimental.

## 14. Engineering Tone

Studio must look and behave like an engineering workbench.

It should avoid:

- fake mission-control visuals
- fake live telemetry screens
- fake command consoles
- unsupported external compatibility claims
- promotional dashboard language
- SaaS/product-platform language before architecture justifies it

The UX must reduce cognitive load without reducing engineering explicitness.

## 15. v0.0 Exit Criteria

The v0.0 charter is complete only when:

1. the repository clearly explains why Studio exists;
2. the repository clearly explains what Studio is not;
3. the relationship with OrbitFabric Core is unambiguous;
4. the Mission Model is explicitly declared as source of truth;
5. Studio is defined as downstream, not authoritative;
6. validation is delegated to OrbitFabric Core;
7. initial UX principles are documented;
8. initial architectural boundaries are documented;
9. project risks are documented;
10. the next milestone is narrow and executable.
